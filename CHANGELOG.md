## Recent Updates (February 2026)

### 23/02/2026 – Honor “no charts” user request

- **Guard rails**: Added explicit rule to Layer 2 (Output Contract Protection): when the user requests no charts, text-only output, or a report without visuals, the agent must not generate charts and must provide the analysis in text and/or tables only. Addresses the issue where the agent generated charts despite user requests like “report without charts”.

### 23/02/2026 – Auto-generated Agent ID

- **Agent ID auto-generation**: Agent ID on the Create Agent page is now generated by the system instead of user-entered. Format: `dashworx_` + random hex suffix (e.g. `dashworx_a1b2c3d4`). IDs are guaranteed unique and never reused, even after an agent is deleted.
- **Backend**: New endpoint `GET /api/agents/generate-id` returns a unique ID. Used IDs are persisted in `used_agent_ids.json` and checked against GCP agents and local profiles. `_persist_used_agent_id()` records IDs when agents are created.
- **Frontend**: Agent ID field is read-only and auto-filled on page load.

### 23/02/2026 – Maximum bytes billed (Create Agent)

- **Maximum bytes billed**: New control on the Create Agent page allowing users to set a per-query bytes limit for BigQuery. If a query would exceed this limit, it fails without incurring a charge. Value 0 means use the project default.
- **Backend**: Forwards `publishedContext.options.datasource.bigQueryMaxBilledBytes` to the GCP API as a string (e.g. `"104857600"` for 100MB).
- **Frontend**: Numeric input with help icon and description text explaining the behaviour.

### 23/02/2026 – Chain of Thought (CoT) – Live streaming, narrative style & UI polish

- **Live CoT streaming**: Chain of Thought now appears step-by-step as the CA API generates it, instead of all at once at the end. Backend uses `stream=True` when calling the CA API, parses incoming messages in real time, and emits SSE events (`cot_step`, `done`, `error`). New endpoint `POST /api/chat/stream` returns Server-Sent Events. Frontend uses `sendChatMessageStream()` to consume the stream and updates the CoT accordion incrementally. Falls back to non-streaming `/api/chat` if streaming fails.
- **Narrative style**: CoT steps are converted from structured labels (e.g. "Schema resolution (question)", "Analysis query") into Google-like narrative text: "I'm resolving the schema and identifying relevant data sources...", "I'm generating the analysis query to answer your question...", etc. via `_to_narrative_cot()` helper. Reasoning/Progress text from the model is passed through unchanged.
- **Generated SQL hidden from CoT**: The "Generated SQL" step is no longer shown in the CoT accordion; users see SQL in the SQL panel instead. Removed from both `_extract_chain_of_thought` and `_extract_cot_step_from_message`, and from the fallback CoT-from-SQL logic.
- **UI polish**: (1) Removed border from the assistant (receiving) message box to match the sender's clean look. (2) CoT section restyled: grey text (`text-gray-500`) to separate from the main response, no borders, "Show reasoning" / "Hide reasoning" toggle. (3) Empty message box during live streaming hidden; only the CoT accordion and "Calling GCP agent..." status show until the final response arrives.

### 20/02/2026 – Create Agent multi-table support & data source UX improvements

- **Multi-table support**: Create Agent page now allows selecting more than one table from the same project. Users can add multiple table sources via "Add another table" and configure each with its own dataset and table. Each source row includes a remove button (when more than one exists).
- **Bug fix – Dataset selection reverting**: Fixed a bug where selecting a dataset would revert to "Select dataset". The cause was calling `updateSource` twice in `handleSourceDatasetChange` (once for dataset, once to clear table), which overwrote the first state update. Fixed by using a single `setSources` call with the functional updater form.
- **Table selection required**: Removed the "All tables in dataset" option from the Table dropdown to prevent accidental selection. Replaced with "Select table" as the placeholder and made table selection required for each source. Validation now requires project, dataset, and table before creating an agent.

### 17/02/2026 – Organization-level project discovery feature

- **Backend**: Added Cloud Resource Manager API integration (`CRM_API` constant and `_crm_get()` helper function) to list projects under a GCP organization.
- **New endpoint**: `GET /api/bq/organizations/{org_id}/projects` - Lists all active projects under a specified organization ID. Accepts numeric org ID (e.g., "766736075314") and returns project details (id, name, projectNumber, lifecycleState).
- **Frontend**: Added organization ID input field and "Load Projects" button in Create Agent page. When org projects are loaded, displays a dropdown to select projects. Manual project ID input remains available for backward compatibility.
- **Error handling**: Enhanced `_crm_get()` to extract and display detailed Google API error messages, including missing permissions when available. Errors now show actual Google API messages instead of generic 403/404 responses.
- **API function**: Added `listOrgProjects(orgId)` in `frontend/src/utils/api.js` to call the new backend endpoint.
- **Auto-load datasets**: Selecting a project from the organization dropdown automatically loads datasets for that project.

**Errors encountered and resolved:**
- **Error 1**: `Service account lacks permission to list projects in organization 766736075314. Ensure org-level access is granted.`

  - **Resolution**: Granted `roles/browser` (Browser) role to service account at organization level. This role includes `resourcemanager.projects.list` permission required to list projects under an organization.

  - **Cause**: The billing project (`conversationalanalyticsapi`) did not have Cloud Resource Manager API enabled. When using `x-goog-user-project` header, the billing project must have the API enabled even if data lives in other projects.
  - **Resolution**: Enabled Cloud Resource Manager API (`cloudresourcemanager.googleapis.com`) in the billing project `conversationalanalyticsapi` via Google Cloud Console or gcloud CLI. After enabling, waited 1-2 minutes for propagation.

**Billing**: All API calls (listing org projects and querying BigQuery data in those projects) are billed to `conversationalanalyticsapi` (the billing project), regardless of which project the data lives in. This allows querying data from any project in the organization while maintaining centralized billing.

## Backend Changes

### New Endpoints Added

#### Agent Management
- `GET /api/agents` - List all agents from `ca_profiles.json` and GCP
- `GET /api/agents/generate-id` - Generate a unique agent ID (format: `dashworx_<hex>`, never reused)
- `GET /api/agents/{agent_id}` - Get agent details from Google Cloud
- `PATCH /api/agents/{agent_id}/instruction` - Update agent system instruction
- `PATCH /api/agents/{agent_id}/label` - Update agent custom label
- `POST /api/agents` - Create new Data Analytics Agent in GCP
- `DELETE /api/agents/{agent_id}` - Delete Data Analytics Agent from GCP
- `PATCH /api/profiles/{profile_key}` - Update profile label
- `DELETE /api/profiles/{profile_key}` - Remove profile

#### Data Sources
- `GET /api/sources` - List data sources from GCP and/or `ca_profiles.json`
  - Returns object with `sources` array and `meta` object
  - `meta` contains `gcp_count`, `local_count`, `gcp_status`, `gcp_error`, `total`

#### BigQuery APIs
- `GET /api/bq/projects` - List BigQuery projects
- `GET /api/bq/datasets` - List datasets in a project
- `GET /api/bq/tables` - List tables in a dataset
- `GET /api/bq/organizations/{org_id}/projects` - List projects under a GCP organization (requires Cloud Resource Manager API enabled and org-level `roles/browser` permission)

#### Chat
- `POST /api/chat` - Send chat message
  - Accepts `agent` field (full GCP resource name) or `profile` field (local profile key)
  - Accepts `maxTurns` parameter for conversation turn limits

### Helper Functions Added
- `ga_headers()` - Get headers for Google API requests
- `ga_get(path)` - GET request to Google API
- `ga_patch(path, update_mask, payload)` - PATCH request to Google API
- `ga_post(path, payload, params)` - POST request to Google API
- `ga_delete(path, max_retries, initial_delay)` - DELETE request to Google API with retry logic
- `_bq_get(url, params)` - BigQuery API GET requests
- `_crm_get(url, params)` - Cloud Resource Manager API GET requests (with enhanced error message extraction)
- `_fetch_sources_from_gcp()` - Fetch Data Analytics Agents from GCP
- `_fetch_agents_from_gcp()` - Fetch agents from GCP for Agent Manager
- `_get_agent_display_label(agent_id, agent_path, profile_key)` - Get agent display label with priority
- `_get_cached_gcp_sources()` - Get cached GCP sources
- `_set_cached_gcp_sources(sources, error)` - Cache GCP sources
- `_invalidate_gcp_cache()` - Invalidate the GCP sources cache
- `_find_agent_path_by_id(agent_id)` - Find agent path by agent ID

### Models Added
- `InstructionPatch` - Pydantic model for instruction updates
- `CreateAgentBody` - Pydantic model for agent creation
- `ChatBody` - Enhanced with optional `agent` field and `maxTurns` parameter

### Caching System
- **In-Memory Cache**: `_gcp_sources_cache` for GCP API responses
- **Configurable TTL**: `GCP_CACHE_TTL` environment variable (default: 300 seconds)
- **Cache Invalidation**: Manual invalidation after agent creation/deletion

### Error Handling
- **Detailed Error Extraction**: Extracts specific error details from GCP API JSON responses
- **Error Status Codes**: Captures HTTP status codes and error reasons
- **Error Messages**: Returns detailed error messages in API responses
- **Logging**: Comprehensive logging for GCP fetch operations and errors
- **Retry Logic**: Exponential backoff retry for permission errors (403) in delete operations
- **Soft-Delete Handling**: Gracefully handles SOFT_DELETED state in delete operations

### Security Enhancements
- **CORS Hardening**: 
  - Environment-defined `ALLOWED_ORIGINS`
  - Restricted `allow_methods` to `GET`, `POST`, `PATCH`, `DELETE`
  - Restricted `allow_headers` to `Content-Type`, `Authorization`, `X-Requested-With`
  - `max_age=3600` for preflight caching
- **Security Headers Middleware**:
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `X-XSS-Protection: 1; mode=block`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - `Strict-Transport-Security: max-age=31536000; includeSubDomains`
  - `Content-Security-Policy`: Comprehensive CSP with restricted sources
- **Input Validation**:
  - Regex validation for `agent` and `profile` paths
  - Length validation for `message` (max 5000 characters)
  - Type and range validation for `maxTurns` (1-20)
  - Array validation for `history`

### Issue 47: Organization Project Listing Permission Error
**Problem**: `Service account lacks permission to list projects in organization 766736075314. Ensure org-level access is granted.` (403 error)  
**Cause**: Service account had BigQuery and Gemini roles at project and organization level, but lacked Resource Manager permissions (`resourcemanager.projects.list`) required to list projects under an organization via Cloud Resource Manager API.  
**Solution**: 
- Granted `roles/browser` (Browser) role to service account at organization level
- This role includes `resourcemanager.projects.list` permission needed for the `/api/bq/organizations/{org_id}/projects` endpoint
- Minimal required role for listing projects under an organization

### Issue 48: Cloud Resource Manager API Not Enabled
**Problem**: `Cloud Resource Manager API has not been used in project conversationalanalyticsapi before or it is disabled.`  
**Cause**: The billing project (`conversationalanalyticsapi`) did not have Cloud Resource Manager API enabled. When using `x-goog-user-project` header to specify billing, that project must have the API enabled even if the actual data/resources are in other projects.  
**Solution**: 
- Enabled Cloud Resource Manager API (`cloudresourcemanager.googleapis.com`) in billing project `conversationalanalyticsapi`
- Can be enabled via Google Cloud Console API Library or gcloud CLI: `gcloud services enable cloudresourcemanager.googleapis.com --project=conversationalanalyticsapi`
- Waited 1-2 minutes for API enablement to propagate before retrying

### Issue 49: Generic Error Messages for Resource Manager API
**Problem**: Backend was returning generic 403/404 error messages instead of detailed Google API error messages, making troubleshooting difficult.  
**Cause**: `_crm_get()` helper was raising HTTPException with raw `r.text` without parsing Google's structured error JSON format.  
**Solution**: 
- Enhanced `_crm_get()` to parse Google API error JSON responses
- Extracts `error.message` from Google's error structure
- Attempts to extract missing permission from `google.rpc.ErrorInfo` metadata when available
- Appends missing permission to error message if found: `"{error_message} (Missing permission: {permission})"`
- Provides actionable error messages instead of generic status codes
